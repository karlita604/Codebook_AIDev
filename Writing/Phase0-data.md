# Phase 0 — Data

## Data filtering (`src/phase0/PRfilter.py`)

`PRfilter.py` pulls three tables from `hao-li/AIDev` (pull requests, repositories, users), joins them on `repo_id` / `user_id`, and applies whichever filters are passed in to narrow the full dataset down to a candidate PR list. Filters available:

- `rejected` (default `True`) — closed but not merged
- `star_minimum` (default `100`) — repo stargazer count
- `language` (default `None`) — GitHub Linguist language name(s), comma-separated
- `agents` (default `None`) — coding agent(s), e.g. `Claude_Code`, `Copilot`, `Cursor`, `Devin`, `OpenAI_Codex`
- `user_minimum` (default `None`) — PR author's follower count
- `forks_minimum` (default `None`) — repo fork count
- `age_minimum` / `age_maximum` (default `None`) — days since PR creation
- `require_body` (default `True`) — excludes PRs whose body is null or whitespace-only

It can be run as a CLI (`python PRfilter.py --star-minimum 500 --language "Python, C, C++"`) or imported as a function (`from PRfilter import filter_prs`). Running it as a script prints the matching PR ids to stdout and writes them to a CSV in `results/phase0/`, named `<MM>-<DD>-<filters>-<count>.csv` (e.g. `07-20-500-pyccpp-1110.csv`, or `07-20-default-7312.csv` when no filter is overridden) — so every filtered pull is reproducible and dated.

Approach: I first tried langid (a lang-detection library) but it misclassified ~20% of genuine English PR titles as French/German/etc. on spot checks — short, jargon-heavy titles like "Fix joypad reads during DMA" or "Add dashboard sharing" are just too ambiguous for statistical language models. Instead I used a script-based heuristic: a title is flagged as non-English only if a large share of its actual letters belong to a non-Latin script (Chinese, Japanese, Korean, Cyrillic, Arabic, Greek, etc.). Checked against the full ~933k AIDev titles, this flags only ~0.9% — and every sampled flagged title was genuinely non-English. Emoji and accented Latin letters (é, ü, ñ) never count against a title, satisfying the "may contain emojis" requirement. No new dependency needed (uses stdlib unicodedata).

filter_prs(..., english_title=True) — new default-True param
CLI: --no-english-title to disable it
Filename descriptor tag alllangs when disabled, so output CSVs stay distinguishable

Similarly, `require_body=True` (new default-True param) drops PRs with a null or
whitespace-only body. CLI: `--no-require-body` to disable it; filename descriptor
tag `allbodies` when disabled.

# Phase 1 — Metadata

## Metadata compilation (`src/phase0/phase1.py`)

`phase1.py` takes the PR-id csv produced by `PRfilter.py` (default input:
`results/phase0/07-21-500-pycsharp-1398.csv`) and joins each id against
`all_pull_request.parquet` to build a short metadata dataframe, one row per PR:

- `id`, `title`, `body`, `agent`, `state`, `created_at`, `closed_at`, `merged_at`,
  `repo_id`, `repo_url`, `html_url`

All of these columns live directly on the PR table, so this phase only needs
`all_pull_request.parquet` — no join against the repository or user tables. The join
is a left merge keyed on `id`, so the input csv's row order and count are preserved;
any id with no match in the PR table would come back with `NaN` metadata rather than
being silently dropped (this hasn't happened in practice — every id from a
`PRfilter.py` run has matched).

It can be run as a CLI (`python phase1.py --input path/to/prs.csv`) or imported as a
function (`from phase1 import build_metadata`). Running it as a script writes
`phase1_<inputfilename>.csv` next to the input csv — e.g.
`results/phase0/07-21-500-pycsharp-1398.csv` →
`results/phase0/phase1_07-21-500-pycsharp-1398.csv`.

This metadata csv is the input to Phase 1.5 (`metrics.py`), which will add the actual
metric columns (`rejected`, `num_diffhunks`, `ngrams`, `interaction_count`, etc. — see
that file's column list) on top of it.
